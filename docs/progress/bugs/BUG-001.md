## 描述

MS-002 整合验证时，Tailscale 活体采集失败。`TailscaleCollector._fetch_status()`
通过 Unix socket 调用 localapi `/localapi/v0/status` 时，tailscaled (1.50+) 返回
`403 Forbidden` / `invalid localapi request`，导致 collect() 抛异常、collector_run
降级为 `down`，面板拿不到任何真实节点。

单元测试因 mock 了 `aiohttp.ClientSession`（不校验 Host / headers）全部通过，掩盖
了该集成缺陷；只有 `@pytest.mark.integration` 的 `test_live_smoke` 在真实 socket
机器上才暴露。

## 复现步骤

1. 在装有 tailscale 1.50+ 且 tailscaled.sock 存在的机器上启动 panel。
2. 触发 `POST /api/tailscale/refresh` 或等待采集周期。
3. `GET /api/tailscale/nodes` 返回空；`collector_run` 中 tailscale 状态为 `down`。
4. 直接 curl 复现：
   `curl --unix-socket /var/run/tailscale/tailscaled.sock http://local-tailscaled/localapi/v0/status`
   → `invalid localapi request`。

## 期望行为

采集器解析出 tailnet 全部真实节点（muxrpi 等）并入库，`/api/tailscale/nodes` 与
`tailscale status` 一致。

## 实际行为

localapi 拒绝请求（403 / invalid localapi request），无节点入库。

## 根因

现代 tailscaled localapi 有两道防护，旧实现都未满足：

1. **Host 必须为 `local-tailscaled.sock`（带 `.sock` 后缀）**。旧代码用
   `http://local-tailscaled`（无 `.sock`），被判 invalid。
2. **必须携带 `Sec-Tailscale: localapi` 请求头**（localapi CSRF / 跨源嗅探防护，
   tailscale ~1.50 起引入）。旧代码不带任何头。

curl 验证：同时满足 `Host: local-tailscaled.sock` + `Sec-Tailscale: localapi` 即返回
200 + 完整 status JSON。

## 修复方案

`src/panel/collectors/tailscale/collector.py`：

- `LOCALAPI_HOST = "local-tailscaled.sock"`，`LOCALAPI_BASE = f"http://{LOCALAPI_HOST}"`
  （aiohttp 从 URL authority 推导 Host，自动带上 `.sock`）。
- 新增 `LOCALAPI_HEADERS = {"Sec-Tailscale": "localapi"}`，`_fetch_status()` 的
  `session.get(...)` 传入 `headers=LOCALAPI_HEADERS`。

修复后 `test_live_smoke` 通过，解析出 9 个真实节点（含 muxrpi=ONLINE）；现有 mock
单测不受影响（mock 不校验 Host/headers），全套 309 passed。

## 后续建议

考虑为单元测试补一条断言：`_fetch_status` 发出的请求 URL host == `local-tailscaled.sock`
且带 `Sec-Tailscale` 头，以防回归（当前 mock 太宽松，无法捕获此类协议层缺陷）。
