## 现象

真机联调中，面板 AI 卡片显示 Codex 用量 **46%**，但该数值实为 **2026-06-17** 那次会话结束时的快照（用户今天 06-28 未使用 Codex，`~/.codex/sessions/2026/06/28` 为空）。面板却标记 `stale=false`，把 11 天前、早已跨越无数个 5h 滚动窗口的过期值当作"当前用量"展示。Claude Code 的重置时间/百分比为本机 jsonl 估算，与官方额度有 15–18 分钟误差。

## 根因

Reporter 各 source 上报时将 `reported_at`（→ 面板 `collected_at`）设为「上报此刻」，而非「数据本身的时间」。面板 ai_usage 的 stale 判定为 `(now - collected_at) > window_seconds * 0.5`，因 `collected_at` 永远是当前时刻，过期数据永远被判为新鲜。Codex「不使用就不写新文件」的特性放大了该问题：reporter 只能取到最近一次会话的旧快照。

## 影响

- 误导性：面板把过期 AI 额度当作实时，用户据此判断会出错（"我没用过怎么 46%"）。
- 非数据丢失/崩溃，但属功能正确性问题 → major。

## 修复

- Codex（及其他从本地缓存反推的源）以**数据自身时间戳**作为 `collected_at`，使面板 stale 判定能正确把过期数据灰掉/标陈旧。
- 超出窗口的过期数据明确判 stale，不伪装为当前值。
- 文档笔误：`tools/reporter/README.md` / `reporter.example.env` 中 `PANEL_URL=...:8000` → `:8080`。
- README 标注 Claude 重置时间/百分比为本地估算（无官方 API），有误差。

对应 TASK（task #13）/ Coder 修复进行中。
