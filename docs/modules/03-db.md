# 模块参考：db（持久化层）

> 模块路径：`src/panel/db/`
> 关联架构：ARCH-001（通用基线）/ ARCH-002（Azure/GPU 专用表）/ ARCH-003（Tailscale）/ ARCH-004（AI 用量）
> 关联任务：TASK-002 / TASK-010 / TASK-020 / TASK-030 / TASK-040 / TASK-016
> 适用读者：需要读写库、新增表/迁移、扩展 Repository 方法的开发者与维护者

---

## 1. 模块概述与职责

`db` 是 Panel Everything 的**持久化层**，是整个应用与 SQLite 之间唯一的 SQL 边界。它解决三类问题：

1. **连接与建表**：用 aiosqlite 打开一个 WAL 模式的长生命周期连接（随 FastAPI lifespan），并在启动时幂等地建好所有表与索引。
2. **统一读写契约**：把所有 SQL 收敛到两个薄 Repository 类（`Repository` 通用三表 + `GpuRepository` Azure/GPU 专用表），上层（采集器、API、SSR、降采样/清理 job）只调方法、不写裸 SQL。方法签名即权威契约，**不得随意更改**。
3. **时间归一化**：库内统一以 ISO8601 UTC 字符串存时间；写库前由 `_iso()` 归一化、读出时由 `_parse_utc()` 还原带 tz 的 `datetime`，消除 naive/aware 混用。

设计取向（来自 ARCH-001「轻量第一」）：单文件 SQLite、零运维、单连接全异步；WAL 提升并发读；表结构刻意泛化——通用三表承载「一个 target 一个标量指标」的数据，富结构（GPU 多卡时序、Tailscale 节点）才进专用表。目标硬件是树莓派（ARM、资源受限），因此所有写都走批量 `executemany` + 单次 `commit`，事件型历史（Tailscale）只在状态变更时落行以省 IO。

## 2. 文件与关键符号清单

### `connection.py` — 连接管理
| 符号 | 职责 |
|------|------|
| `async connect(db_path) -> aiosqlite.Connection` (`connection.py:14`) | 打开并配置连接：文件型路径自动 `mkdir(parents=True)` 父目录（`:memory:` / `file:` 前缀除外，`connection.py:33`）；设 `row_factory = aiosqlite.Row`；执行四条 PRAGMA。调用方负责 `close()`。 |

四条 PRAGMA（`connection.py:40-43`）：`journal_mode=WAL`（并发读）、`synchronous=NORMAL`（WAL 下安全且更快）、`foreign_keys=ON`（外键约束生效，级联删除依赖此项）、`busy_timeout=5000`（锁等待 5s）。

全部 `CREATE ... IF NOT EXISTS`，由 `migrate.run()` 幂等执行。包含 ARCH-001 通用三表 + ARCH-002 五张专用表（见 §3）。注意：**ARCH-002 的五张表写在 `schema.sql` 里**（`schema.sql:47-131`），不在 `migrations/`；只有 ARCH-003/004 的表才以独立迁移文件追加。

## 3. 关键数据结构 / 表 / 契约

### 3.1 ARCH-001 通用三表（`schema.sql:8-45`）

承载「一个 target 一个标量指标」的泛化数据（VM 电源态、节点在线、AI 额度、动态公网 IP 等）。

- **`latest_snapshot`** — 最新快照。PK = `(collector, target_id, metric)`，写为 UPSERT。列：`value_num REAL?`、`value_text TEXT?`、`status`（`ok|unreachable|error`）、`collected_at`、`updated_at`（写库时刻）。`target_id` 无维度时用 `0`。索引 `idx_latest_collector(collector)`。
- **`metric_history`** — append-only 时序，`id` 自增。查询索引 `idx_history_query(collector, target_id, metric, collected_at)`。retention job 按 `collected_at` 删旧（TASK-040）。
- **`collector_run`** — 每次采集运行一行，`status`（`up|down|error`）、`sample_count`、`duration_ms`、`error`（脱敏摘要）、`ran_at`。索引 `idx_run_latest(collector, ran_at DESC)`。

对应行类型 `SnapshotRow` / `HistoryRow` / `CollectorRunRow`（`repository.py:27/39/50`），字段与列一一对应；时间列在行类型里是 **str**（直接透传库内 ISO 串，不解析）。例外：`get_last_success()` 返回的是 `_parse_utc` 后的 `datetime`。

### 3.2 ARCH-002 五张专用表（`schema.sql:52-131`）

- **`servers`** — 服务器注册表。`name UNIQUE`；`ssh_key_path` **只存路径引用，不存私钥内容**；`has_gpu` 用 `INTEGER 0/1`（SQLite 无原生 bool）；`ssh_port` 默认 22，`ssh_user` 默认 `azureuser`。索引 `idx_servers_name`。
- **`azure_vm_status`** — VM 状态快照，`server_id` 为 PK（每台 VM 一行，upsert）。`power_state`（展示值 Running/Stopped/...）、`power_state_raw`（Azure 原始 `PowerState/running`）、`is_running 0/1`。`FOREIGN KEY(server_id) REFERENCES servers(id) ON DELETE CASCADE`。
- **`gpu_metrics`** — GPU 多卡时序，append-only，`id` 自增。每张卡每次采集一行（`gpu_index` 0-based 对应 nvidia-smi 行序）。`mem_pct` 列由写方法计算。两个索引：`idx_gpu_metrics_query(server_id, gpu_index, collected_at)`、`idx_gpu_metrics_server_latest(server_id, collected_at DESC)`。`ON DELETE CASCADE`。
- **`gpu_metrics_5m` / `gpu_metrics_1h`** — 降采样桶（TASK-016 填充）。列结构相同：`avg_util_pct`、`avg_mem_pct`、`max_temp_c`、`max_power_w`、`sample_count`、`bucket_start`（粒度对齐）。各有 **UNIQUE 索引** `(server_id, gpu_index, bucket_start)`，是 upsert 去重键。`ON DELETE CASCADE`。

对应行类型 `ServerRow` / `VmStatusRow` / `GpuMetricRow` / `GpuBucketRow`（`gpu_repository.py:31/47/57/73`）。其中 `has_gpu`/`is_running` 在行类型里被还原为 `bool`，时间列仍为 `str`。`GpuBucketRow` 复用给两张降采样表。

### 3.3 ARCH-003 Tailscale 两表（`migrations/003_tailscale.sql`）

- **`tailscale_nodes`** — 每节点一行，`node_key UNIQUE`（Self/Peer PublicKey，永久标识，upsert 键）。`tailscale_ips TEXT` 存 **JSON 数组**（默认 `'[]'`）；`online_state`（`ONLINE|OFFLINE|LONG_OFFLINE`）；`last_seen_at` 在线时为 NULL。
- **`tailscale_node_events`** — 事件历史，**仅在 `online_state` 变更时 INSERT**（含首次发现 `from_state=NULL, note='first_seen'`），避免树莓派高频写入。

对应行类型 `TailscaleNodeRow` / `TailscaleNodeEventRow`（`repository.py:355/372`）：`tailscale_ips` 已 `json.loads` 为 `list[str]`；`is_exit_node` 还原为 `bool`；时间列已 `_parse_utc` 为 `datetime`（`last_seen_at` 在线时为 `None`）。

### 3.4 ARCH-004 ai_provider 表（`migrations/004_ai_provider.sql`）

静态配置元数据表。`provider UNIQUE`（`codex`/`claude_code`/`chatgpt`）、`display_name`、`source_type`（`local_jsonl`/`oauth_api`/`manual`）、`window_seconds`、`enabled 0/1`。`id` 同时充当 `latest_snapshot`/`metric_history` 里 AI 用量数据的 `target_id`（provider 名 ↔ id 映射）。对应行类型 `AiProviderRow`（`repository.py:594`）。

### 3.5 输入契约（来自其他模块）

- `MetricSample`（`collectors/base.py:23`）：`target_id` / `metric` / `value_num?` / `value_text?` / `status`（`ok|unreachable|error`）/ `collected_at: datetime`。写 `upsert_snapshot` / `append_history` 的入参元素。
- `CollectorResult`（`collectors/base.py:35`）：`name` / `status`（`up|down|error`）/ `sample_count` / `duration_ms` / `error?` / `ran_at`。写 `record_collector_run` 的入参。
- `ServerIn`（`domain/models.py:59`，Pydantic）：`insert_server` 的入参；含 `ssh_key_path`（仅写库，响应模型 `ServerOut` 白名单不回传）。
- `GpuSample`（`gpu_repository.py:95`）：`append_gpu_metrics` 的入参（无 `mem_pct`）。

## 4. 对外接口与调用关系

### 4.1 `Repository` 方法契约（通用三表）

**写**（各自一次 `commit`）：
- `async upsert_snapshot(collector, samples)` (`repository.py:107`) — 按 `(collector,target_id,metric)` 批量 UPSERT；空列表 noop；`updated_at=now`。
- `async append_history(collector, samples)` (`repository.py:145`) — 批量 append；空列表 noop。
- `async record_collector_run(result: CollectorResult)` (`repository.py:172`) — append 一行；`result.error` 须**由调用方脱敏**。

**读**：
- `async get_snapshot(collector) -> list[SnapshotRow]` (`repository.py:193`) — 该 collector 全部快照，按 `target_id, metric`。
- `async get_snapshot_metric(collector, target_id, metric) -> SnapshotRow | None` (`repository.py:207`)。
- `async get_history(collector, target_id, metric, since, until=None, limit=1000) -> list[HistoryRow]` (`repository.py:223`) — 范围 `[since, until]`；内层 `DESC + LIMIT` 取最近 N 条、外层 `ASC` 呈现（保证裁剪到最新数据后仍升序）。
- `async get_last_success(collector) -> datetime | None` (`repository.py:263`) — 最近 `status='up'` 的 `ran_at`（`_parse_utc` 后），供 stale 判定。
- `async get_last_run(collector) -> CollectorRunRow | None` (`repository.py:278`) — 最近一次运行（任意 status，按 `id DESC`）。
- `async get_all_last_runs() -> list[CollectorRunRow]` (`repository.py:293`) — 每 collector 最近一次（子查询 `MAX(id)` JOIN），供数据源状态条。

**注入方法**（见 §6，运行期等同实例方法）：
- `async upsert_tailscale_node(...) -> int`、`async get_all_nodes()`、`async get_node_by_id(id)`、`async get_node_events(node_key, limit=100)`。
- `async get_ai_provider_id(provider) -> int | None`、`async get_ai_providers() -> list[AiProviderRow]`（仅 `enabled=1`，按 id 升序）。
- `async prune_history(before: datetime) -> int` — `DELETE ... collected_at < before`（严格小于），返回删除行数。

### 4.2 `GpuRepository` 方法契约（ARCH-002 专用表）

**servers CRUD**：`get_all_servers()`（`:143`）、`get_server(id)`（`:156`）、`insert_server(ServerIn) -> int`（`:171`，重复 name 抛 `aiosqlite.IntegrityError`）、`delete_server(id) -> bool`（`:204`，`ON DELETE CASCADE` 连带删 vm_status/gpu_metrics）。

**azure_vm_status**：`upsert_vm_status(server_id, power_state, power_state_raw, is_running, collected_at)`（`:218`，按 `server_id` UPSERT）、`get_vm_status_all()`（`:252`）、`get_vm_status(server_id)`（`:264`）。

**gpu_metrics**：`append_gpu_metrics(samples)`（`:280`，批量 append，内部算 `mem_pct`，空列表 noop）、`get_latest_gpu_metrics(server_id)`（`:317`，子查询每卡 `MAX(id)`，每卡最新一行）、`get_gpu_history(server_id, gpu_index, since, until=None, limit=1000)`（`:343`，内 DESC+LIMIT / 外 ASC）。

**降采样读写**（TASK-016）：`upsert_5m_bucket(GpuBucketRow)` / `upsert_1h_bucket(...)`（`:386/390`，经 `_upsert_bucket` 走 `INSERT OR REPLACE`，UNIQUE 索引去重）；`get_gpu_history_5m(...)` / `get_gpu_history_1h(...)`（`:420/433`，默认 `limit=200`，内 DESC+LIMIT / 外 ASC）。
**降采样聚合源**：`aggregate_raw_buckets(since, until)`（`:487`，从 `gpu_metrics` 聚合，**仅 `status='ok'`** 行，避免 unreachable/error 的 NULL 污染均值）；`aggregate_5m_buckets(since, until)`（`:499`，5m 桶聚成 1h，`sample_count` 用 `SUM`）。两者返回 `list[tuple]`：`(server_id, gpu_index, avg_util, avg_mem, max_temp, max_power, count)`。
**retention 清理**：`delete_raw_metrics_before(cutoff)`（`:546`）、`delete_5m_buckets_before(cutoff)`（`:555`），返回删除行数。

## 6. 扩展点（可操作步骤）

### 6.1 新增一张模块专用表（迁移）
1. 在 `migrations/` 新建 `005_<模块名>.sql`（编号紧接现有最大号）。
2. 所有语句用 `CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS`；若需种子数据用 `INSERT OR IGNORE` 且**时间戳写字面量常量**（不要用 `CURRENT_TIMESTAMP`/动态值，否则每次启动破坏幂等，参考 `004_ai_provider.sql`）。
3. 外键引用 `servers` 时加 `ON DELETE CASCADE`（依赖 `foreign_keys=ON`）。
4. 无需改 `migrate.py`——`_read_migrations()` 自动按文件名升序拾取。`migrate.run()` 启动幂等执行。
5. 写一个 `tests/test_<模块>_schema.py`，断言表/索引存在、`migrate.run()` 多次不报错、UNIQUE/CASCADE 约束生效（参考 `tests/test_gpu_schema.py`）。

### 6.2 给 `Repository` 新增一个方法（setattr 注入模式）
`Repository` 类体已被「封口」（Tailscale/AI/retention 扩展不改类体）。新增方法的既定写法（见 `repository.py:384-660`）：
1. 在模块**类体之后**定义 `async def _my_method(self: Repository, ...) -> ...:`，`self` 显式标注为 `Repository`，内部用 `self._conn`。
2. 需要的话同区域定义新的 `@dataclass(slots=True)` 行类型 + `_to_xxx(row)` 映射函数。
3. 文件末尾 `Repository.my_method = _my_method  # type: ignore[attr-defined]`。
4. 调用方处加 `# type: ignore[attr-defined]`（静态检查器看不到注入方法），如 `collectors/tailscale/collector.py:181`、`api/tailscale/routes.py:58`。
> 说明：注入是历史约定（避免在已审定的类体封口处改动）；新代码若位于同一文件且无封口顾虑，更整洁的做法是直接写进类体。`GpuRepository` 的方法即全部在类体内声明，可作正例。

### 6.3 新增一个采集器写入的指标
- **标量指标**（如新增一个 provider 状态）：直接复用通用三表——采集器产出 `MetricSample`（选好 `collector` 名 + `target_id` + `metric`），调 `upsert_snapshot` + `append_history`。无需改 db。动态公网 IP（ARCH-002 Addendum）即用此法：`metric="public_ip"`、`value_text=<IP>`，不新增列、不新增表。
- **富结构时序**（多维/多列）：新增专用表（§6.1）+ 在合适的 Repository 加方法（§6.2 或直接进 `GpuRepository` 类体），不要硬塞进通用三表。

### 6.4 新增一个清理/降采样 job
- 在对应 Repository 加 `delete_*_before(cutoff) -> int` 或 `aggregate_* / upsert_*_bucket` 方法。
- job 函数放 `collectors/`（如 `retention.py` / `gpu/downsampler.py`），在 `main.py` lifespan 用 `scheduler.add_job(...)` 注册（参考 `main.py:82-103`）。

### 6.5 新增对外端点/响应模型
db 层只产 dataclass 行类型；对外 JSON 必须经 `domain/models.py` 的 Pydantic 白名单响应模型（基类 `PublicModel`）。新增端点时：行类型 → 白名单模型转换在 API 层做，**凭证字段（如 `ssh_key_path`）一律不进响应模型**。

## 9. 注意事项 / 降级语义 / gotchas

- **单连接共享**：`Repository` 与 `GpuRepository` 共用 lifespan 的同一 `aiosqlite.Connection`。无连接池，WAL 提供并发读。写方法各自 `commit`，无显式跨方法事务。
- **`foreign_keys=ON` 是 CASCADE 的前提**：连接级 PRAGMA，每次新连接都要设；`connection.connect()` 已设。删 server 自动连带删 vm_status/gpu_metrics（及其它引用 servers 的表）。
- **时间归一化双向**：写库前必经 `_iso()`（naive 视为 UTC），读出时 `get_last_success` / Tailscale 行经 `_parse_utc` 还原带 tz datetime；但通用/GPU 行类型的时间列**保持库内 str 原样不解析**，由上层按需处理。ISO8601 UTC 字符串**字典序 == 时间序**，所以 `prune_history` / `delete_*_before` 用字符串 `<` 比较即正确（`repository.py:643` 注释）。
- **`mem_pct` 由写方计算**：`GpuSample` 无 `mem_pct`；`append_gpu_metrics` 在 `mem_used_mib` 非 None 且 `mem_total_mib` 真值时算 `round(used/total*100, 2)`，否则存 NULL（`gpu_repository.py:293-297`）。聚合 `aggregate_raw_buckets` **只算 `status='ok'` 行**，避免 unreachable/error 的 NULL 列污染均值（`gpu_repository.py:528-539`）。
- **降采样 upsert 用 `INSERT OR REPLACE`**：REPLACE 会换新自增 `id`，但桶以 UNIQUE 索引 `(server_id, gpu_index, bucket_start)` 去重，`id` 非业务键（`gpu_repository.py:394-399`）。
- **Tailscale 事件低写入**：`upsert_tailscale_node` 先查旧 `online_state`，**仅在状态变更**（或首次发现）时写 `tailscale_node_events`，节省树莓派 IO；返回 `tailscale_nodes.id`。`tailscale_ips` 在库里是 JSON 字符串，写时 `json.dumps`、读时 `json.loads(... or "[]")`。
- **setattr 注入对静态检查不可见**：注入方法在 `Repository.xxx = _xxx` 处与调用处都需 `# type: ignore[attr-defined]`。删除/重命名注入方法时记得同步所有调用点。
- **`migrations/` 编号即顺序**：按文件名 `sorted()` 升序执行；新增迁移必须编号大于现存最大号，否则会插到中间打乱预期顺序（对纯 IF NOT EXISTS 通常无害，但种子数据/依赖顺序的迁移会受影响）。
- **种子数据幂等**：`004_ai_provider.sql` 用 `INSERT OR IGNORE` + **字面量时间戳常量**；若改用动态时间会让 `executescript` 每次启动产生差异，破坏「重复调用无副作用」的契约。
- **`schema.sql` vs `migrations/` 划界**：ARCH-001 通用三表 + ARCH-002 五表都在 `schema.sql`；ARCH-003/004 才用迁移文件。这是历史划分（ARCH-002 表随基线一起落地），新增模块一律走 `migrations/`，不要再往 `schema.sql` 追加。
- **凭证白名单**：`servers.ssh_key_path` 只存路径、`ServerRow` 内部保留供采集器读取，但 API 响应模型 `ServerOut`（`domain/models.py`）白名单**不含 `ssh_key_path`**，不回传前端（ARCH-001 凭证管理规范 / ARCH-002）。`collector_run.error` 写库前须由调用方脱敏。
- **e-ink / 无认证背景**（影响 db 设计取向，非 db 代码直接约束）：单用户、低频访问、无鉴权（依赖 Tailscale 内网），所以无并发写竞争压力、无审计表；历史用 append-only + 降采样/retention 控制磁盘。
- **`insert_server` 边界**：`lastrowid is None` 时抛 `RuntimeError`（`gpu_repository.py:200`，`# pragma: no cover`）；重复 name 直接冒泡 `aiosqlite.IntegrityError`，由 API 层转 409/400。
