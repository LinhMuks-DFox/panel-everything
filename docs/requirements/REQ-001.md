## 背景

Panel Everything 是一个**单用户**的轻量级设备监控面板。系统部署在树莓派上，仅供一位用户使用（以后也不会增加用户），需要支持容器化部署以便维护和迁移。由于使用频率不高、用户量仅一人，整体设计必须以"轻量"为第一原则，避免不必要的资源消耗。

系统通过 Tailscale 内网隔离实现访问控制，不需要用户认证/登录机制。

本需求定义整个系统的非功能约束和部署要求，作为所有其他功能需求的前提条件。

## 用户故事

- 作为唯一用户，我希望通过容器一键部署整个系统，以便快速搭建和迁移。
- 作为唯一用户，我希望系统在树莓派上稳定运行且资源占用极低，以便树莓派可以同时运行其他服务。
- 作为唯一用户，我希望在 Kindle/iPhone/iPad 等设备上都能正常访问面板，以便随时随地查看状态。
- 作为唯一用户，我希望不需要登录就能直接访问面板，因为系统已经通过 Tailscale 内网隔离保护。

## 功能需求

- [ ] 系统支持通过容器（如 Docker Compose）一键部署
- [ ] 提供容器镜像构建配置，支持 ARM 架构（树莓派）
- [ ] 前端为 Web 应用，响应式设计适配多种屏幕尺寸
- [ ] 前端需适配 Kindle e-ink 屏幕（低刷新率、灰度显示）
- [ ] 前端需适配 iPhone 竖屏和 iPad 横屏/竖屏
- [ ] 系统应提供基本的健康检查端点

## UI 交互原则

以下原则适用于所有功能模块的前端设计：

- **单屏总览**: 所有核心信息在首屏一眼可见，不需要滚动或切换页面就能获取全局状态
- **零点击获取**: 关键数据直接展示在面板表面，不藏在弹窗、下拉菜单或二级页面里
- **信息密度优先**: 在不牺牲可读性的前提下，最大化单屏信息密度
- **状态一目了然**: 使用颜色、图标等视觉编码让异常状态（离线、额度告急、GPU 满载）瞬间可辨
- **被动消费**: 面板是"看"的，不是"用"的 — 日常使用场景是打开就看，不是操作

## 访问控制

- **认证方式**: 不需要用户登录/认证机制。系统依靠 Tailscale 内网隔离实现访问控制 — 只有在同一 Tailscale 网络中的设备才能访问面板。
- **用户模型**: 单用户系统，不需要用户管理、多租户、权限系统。
- **无需认证中间件**: 后端 API 无需任何认证/鉴权中间件。

## 数据持久化

- **历史数据**: 系统需要持久化存储历史监控数据，以支持趋势图等历史数据展示功能。
- **具体存储方案**: 由 Architect 根据系统约束（树莓派资源、数据量）选择合适的持久化方案。

## 非功能需求

- **性能**: 系统空闲时内存占用应控制在合理范围内（树莓派通常 1-4GB RAM）；CPU 占用在无访问时应趋近于零
- **兼容性**: 前端兼容主流浏览器（Safari/Chrome）以及 Kindle 实验性浏览器；后端容器需支持 linux/arm64 和 linux/amd64
- **可维护性**: 容器化部署，支持通过环境变量或配置文件进行外部配置
- **可用性**: 系统不需要高可用，单实例部署即可；但应具备自动重启能力（容器 restart policy）
- **安全性**: 依靠 Tailscale 内网隔离，不需要额外的认证机制。面板仅在 Tailscale 网络内可访问。
- **用户规模**: 单用户系统，无需考虑多用户并发、用户管理、权限隔离等问题

## 验收标准

- [ ] 执行 `docker compose up` 可在树莓派上成功启动全部服务
- [ ] 系统空闲时总内存占用符合树莓派资源限制（具体阈值待定）
- [ ] 在 iPhone Safari 上可正常访问和操作面板
- [ ] 在 iPad Safari 上可正常访问和操作面板
- [ ] 在 Kindle 浏览器上可访问面板并正确显示核心信息
- [ ] 容器镜像支持 ARM64 架构

## 开放问题

- 树莓派的具体型号和可用内存/存储？（影响资源限制的具体数值）
- 是否需要 HTTPS？如果需要，证书如何管理（Let's Encrypt / 自签名 / Tailscale HTTPS）？
- Kindle 适配的优先级如何？是 P0（必须完美适配）还是 P2（能看就行）？
- "不是一直都用" —— 是否意味着可以使用按需启动（如通过 cron 定时启停容器）来进一步节省资源？
- 数据刷新策略：轮询间隔多少？是否需要 WebSocket 实时推送？
- 历史数据保留策略：保留多久的历史数据？（影响存储需求）
